DeepSeek-V4-Flash 在 Ada 架构上的部署：SM89 适配实践

Created2026-05-21|Updated2026-05-21|技术实践
|Post Views:

背景

DeepSeek-V4-Flash 是 158B 参数的 MoE 模型。它的官方推理管线依赖 Hopper（SM90）专用特性——DeepGEMM、TMA、WGMMA。如果显卡架构低于 SM90，启动直接崩溃。

目标是在 RTX 4090（SM89，Ada Lovelace） 上跑通这个模型。

问题拆解

启动崩溃的原因有四个：

1. DeepGEMM 算子无法运行。 DeepGEMM 是 DeepSeek 自研的高性能 GEMM 库，只用 Hopper 指令集。SM89 上所有调用 DeepGEMM 的地方都会抛 NotImplementedError。

2. 自定义算子和硬件不兼容。 模型中有一个叫 Multi-head Hyper-Connection 的算子依赖 TileLang 编译管线，TileLang 又依赖 DeepGEMM，链条上任何一个环节不支持 SM89 就跑不了。

3. FP8 量化格式的兼容问题。 模型权重使用了 UE8M0 格式的 scale 因子，Triton 编译器无法直接处理这种数据类型，导致加载 kernel 时崩溃。

4. 并行策略有隐藏约束。 模型的 O 投影层要求张量并行度不超过 8，否则 tensor shape 会算错。

这四个问题串在一起，任何一环不解决模型都起不来。

解决思路

方案的特点可以总结为三件事：

一是全局旁路。 DeepGEMM 的入口加一个架构检测开关，SM89 上自动走 PyTorch reference kernel 替代 DeepGEMM。代价是性能有损失，但能保证推理路径完整。

二是算子替换。 MHC 算子用纯 Triton 重写。Triton 是 Python 写的 kernel 编译器，不依赖硬件架构。原来 TileLang 编译的 kernel 无法在 SM89 上编译，换成 Triton 版本就能正常跑了。

三是数据格式适配。 在调用 Triton kernel 之前，先把 UE8M0 格式的 scale 因子转成 FP32，再传入 kernel。相当于在数据入口加了一个翻译层。

这三个改动覆盖了崩溃的全部根因。适配范围涉及 18 个源文件，以静态补丁包的方式注入容器。

性能基线

修复启动崩溃后，模型可以正常推理了。当前性能：

硬件 并行配置 吞吐量
16 卡节点（RTX 4090 桌面版 × 16） TP=8, DP=2 ~3.2 tok/s
32 卡节点（RTX 4090 Laptop × 16GB × 32） TP=8, PP=2, DP=1 ~2.1 tok/s

作为参考，H100 上同模型的官方测速是 30+ tok/s。差距不仅来自算力差异——RTX 4090 没有 NVLink，跨卡通信走 PCIe，带宽不到 NVLink 的十分之一；也没有 FP8 tensor core，FP8 矩阵乘在 SM89 上要回退到 FP32 模拟，吞吐折半。

3.2 tok/s 不代表终点。 后面的优化工作围绕几个方向：MTP 投机解码（让一次推理产出多个 token）、CUDAGraph 捕获（消除 kernel launch 调度开销）、以及手工调优的 Triton kernel 替换热点算子。这部分下次再展开。

三条经验

  1. 架构门控比动态打补丁靠谱。 早期的方案用 monkey-patch 动态替换模型层的 forward 方法，结果触发了 PyTorch 的 Graph Break，调试成本反而更高。在算子入口用架构版本做条件分支，不破坏编译图，是最干净的方案。

  2. 容器部署选静态文件注入。 18 个文件全部用 cp -r 复制进容器，不依赖 pip install 后的 patch 钩子。回退时换一套文件就行，可复现性好。

  3. 性能天花板要先跑通再说。 启动都没过的时候谈 optim，没用。先解决掉所有 crash，拿到一个可测量的基线，才能知道瓶颈在哪、该怎么优化。
